# 05 · インシデント対応手順

## 1. 重大インシデントの定義

以下のいずれかを **重大インシデント** として扱います。

| レベル | 定義 | 対応 SLA |
|---|---|---|
| **P0** | 全 Buyer のデータ漏えい可能性 / API 完全停止 / 不正な課金発生 | **初動 1 時間以内**, 恒久対応 24 時間以内 |
| **P1** | 一部 Buyer のデータ漏えい / Kill Switch が効かない / Pay Token 署名鍵漏えい疑い | **初動 4 時間以内**, 恒久対応 48 時間以内 |
| **P2** | 単一 Buyer の認可バグ / 課金額の誤差 / ログ欠損 | **初動 24 時間以内**, 恒久対応 7 日以内 |
| **P3** | UI バグ / ドキュメント誤記 / 軽微な性能低下 | ベストエフォート |

## 3. インシデント シナリオ別 プレイブック

### シナリオ A: Buyer JWT が漏えいした

**Buyer 側の即時対応:**
1. `lemoncake.xyz/settings` から **全 Buyer JWT を revoke**（1 クリック）
2. 漏えい範囲を特定するため、監査ログを参照（過去 48 時間の API 呼び出し）
3. 不正な課金があれば `revoke_token` で該当 Pay Token を停止

**LemonCake 側の対応:**
- contact@aievid.com へ通報 → 30 分以内に担当者が対応開始
- Buyer ID 単位で **全 Pay Token を一括 revoke** 可能
- 必要であれば Buyer の API 署名鍵を差し替え（全 Pay Token 無効化）

### シナリオ B: Kill Switch ボタンが効かない

- ダッシュボードで revoke しても該当 Pay Token が課金を通してしまう → **P1**
- 対応: プロキシ側の revoke list キャッシュを強制リフレッシュ（オペレーター操作、< 5 分）
- 原因分析: `audit_logs` から revoke 操作のタイムスタンプを確認し、同時刻の課金リクエストと比較

### シナリオ C: 署名鍵漏えい疑い

- **P0**、直ちに全サービス停止
- 新しい鍵ペアを生成し、JWK セットを切り替え（< 60 秒で既存 Pay Token を全て無効化）
- 全 Buyer に email 通知、ダッシュボードに警告バナー表示
- 48 時間以内にポストモーテム公開

### シナリオ D: freee / QuickBooks 連携障害

- 原因: OAuth リフレッシュ失敗 / 上流 API 停止
- 対応: 仕訳キューに蓄積、復旧後再送
- Buyer 通知: メール + ダッシュボード上でキュー件数表示
- SLA: **課金処理自体には影響なし**（仕訳は非同期）

## 4. ポスト モーテム

P0 / P1 発生時は **96 時間以内に公開ポストモーテム** を以下に公開:

- https://github.com/evidai/lemon-cake/issues （ラベル: `post-mortem`）
- email で関連 Buyer に通知
- 再発防止策と期限付きアクション アイテムを含む

## 5. 導入企業側でやっておくべき準備

1. **Buyer JWT のローテーション ポリシー**: 四半期ごと推奨
2. **Dify 側の監査ログ保存**: Dify ワークスペース設定で保存期間を有効化
3. **連絡先の共有**: 稟議書で自社側の緊急連絡先を LemonCake と共有
4. **テーブルトップ演習**: 半年に 1 回、Kill Switch の発動訓練
